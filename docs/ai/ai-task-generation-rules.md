# AI Task Generation Rules

This document defines how AI assistants should generate tasks and instructions
for the DevOps Lab.

Its goal is to ensure that instructions are clear, deterministic, and aligned
with DevOps engineering practices.

---

# Task Structure

Each task generated by the AI should include:

1. Objective
2. Concept Explanation
3. Step-by-Step Instructions
4. Expected Result
5. Verification Step

This structure ensures that tasks are both educational and actionable.

---

# Concept Before Action

Before providing commands or code, the AI should briefly explain
the concept being introduced.

The explanation should be concise and focused on understanding
the engineering principles behind the task.

---

# Deterministic Instructions

All instructions must clearly specify:

• which directory commands should be executed in
• which files will be created or modified
• whether the file must be edited or fully rewritten
• the expected outcome

Ambiguous instructions must be avoided.

---

# Small Incremental Tasks

Tasks should be incremental.

Each task should introduce only one or two new concepts at a time.

Avoid large tasks that combine multiple unrelated topics.

---

# Repository Awareness

Before generating instructions that modify files, the AI must:

1. Request the file content
2. Review the existing structure
3. Propose minimal changes

The AI must never assume the contents of repository files.

---

# Avoid Repository Pollution

The AI should avoid creating unnecessary files.

Whenever possible, existing documentation should be extended instead of
creating new documents.

---

# Phase Alignment

Tasks must align with the phase progression defined in:

docs/lab-phase-map.md

The AI must not introduce concepts from later phases prematurely.

---

# Debugging Discipline

When troubleshooting issues:

1. Request logs or command output
2. Request configuration files
3. Analyze the problem before suggesting fixes

The AI must avoid speculative debugging.

---

# DevOps Mindset

Tasks should encourage real DevOps practices such as:

• reproducible workflows
• clear documentation
• incremental improvement
• debugging methodology
• automation thinking

The goal is to develop engineering habits, not just complete exercises.

## Safe Command Generation Rules

When generating shell commands, the AI must avoid destructive or non-deterministic patterns.

Forbidden patterns:

mv *
rm *
cp *
docker system prune -a
git reset --hard

Unsafe wildcard operations must never be generated.

Instead the AI must use deterministic patterns such as:

mv *.zip artifacts/
mv artifact-* artifacts/

Commands must never affect the entire repository unless explicitly required and clearly explained.

All commands must be safe to execute in a learning environment.

# Wildcard Safety

Wildcards must only be used when the file pattern is explicitly defined.

Unsafe patterns:

mv *
cp *
rm *

Safe patterns:

mv *.zip artifacts/
mv report-*.txt reports/
cp build-artifact-* artifacts/

Commands must never move or delete the entire working directory.

## YAML Parsing Rules

When inspecting YAML files the AI must avoid generic patterns that match any key.

Commands such as:

grep "^\s*[a-zA-Z_]*:" file.yml

should not be used because YAML contains many keys that are not relevant.

Instead the AI must target specific sections such as:

jobs
steps
uses
runs-on

Example:

grep -A 10 "^jobs:" .github/workflows/*.yml


# Inspection First Rule

Before performing system modifications, the AI should generate
inspection tasks.

Inspection tasks include:

• examining Docker images
• analyzing container metadata
• reviewing pipeline configuration
• inspecting logs
• checking build artifacts
• analyzing CI/CD executions

The student should understand the system state before modifying it.

# Engineering Value Rule

Tasks must provide real engineering value.

The AI must avoid generating tasks that only repeat simple commands
without introducing new concepts or meaningful analysis.

Examples of low-value tasks to avoid:

• repeating the same command multiple times
• copying files without purpose
• generating artificial output files that are not used later

Every task should contribute to at least one of the following:

• deeper system understanding
• debugging skills
• system inspection
• reproducibility verification
• security awareness
• pipeline reliability

# Analysis Requirement

Whenever possible, tasks should include an analysis step.

Instead of only verifying that a command worked, the student should
be encouraged to interpret the result.

Examples:

• analyze why a container layer exists
• explain why two builds produced different hashes
• inspect why a pipeline step failed
• understand why a vulnerability appears in an image

The goal is to develop debugging and system reasoning skills.

# Command Precision Rule

Commands generated by the AI must target specific information.

Generic parsing commands that generate excessive noise must be avoided.

Examples of bad commands:

grep ":" file.yml
grep ".*" file
ls -R /

Commands should extract meaningful information related to the task.

Examples of good commands:

grep "^FROM" Dockerfile
grep "HEALTHCHECK" Dockerfile
grep -A 10 "^jobs:" .github/workflows/*.yml
docker inspect container_name


# Task Size Limit

Tasks should remain reasonably scoped.

A single lab day should typically contain:

• 3 to 12 tasks
• each task focusing on one concept

Extremely long sequences of commands without explanation should be avoided.

The goal is clarity and depth rather than volume.
